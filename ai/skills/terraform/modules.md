# Modules

A module is a directory of `.tf` files that gets called by another directory. That's it — every Terraform directory is a "module" technically, the question is just whether anything calls it. In practice, modules come in two flavors:

| Flavor | Lives under | Purpose |
|---|---|---|
| **Root / composition root** | `environments/<cloud>/<env>/` | The thing you run `tofu apply` against. Wires together child modules and resources to describe one environment. |
| **Child module** | `modules/<name>/` (or external registry/git) | A reusable unit. Doesn't configure providers, doesn't have a backend, doesn't get applied directly. Has multiple call sites. |

The most common AI failure mode is over-modularizing: wrapping a single `aws_s3_bucket` in a "bucket module" that the caller passes ten args to, when inlining the resource would be three lines. A module is justified when (a) the boundary is meaningful (a VPC, a service, a database cluster — something a human would name) **and** (b) there are 2+ real call sites or there will be soon. Otherwise: inline.

## What goes in a module

```
modules/<name>/
├── versions.tf      # required_providers (NO version pin — caller controls)
├── variables.tf     # all inputs, fully typed, with validation
├── main.tf          # resources, locals, data sources
├── outputs.tf       # everything callers might need
├── README.md        # auto-generated by terraform-docs
└── tests/
    └── basic.tftest.hcl
```

For larger modules, split `main.tf` into topic files (`network.tf`, `iam.tf`, `compute.tf`). HCL doesn't care — files in a directory are concatenated at parse time. Don't introduce subdirectories inside one module; HCL doesn't recurse.

### `versions.tf` in a module

```hcl
terraform {
  required_version = ">= 1.10.0"

  required_providers {
    aws = {
      source                = "hashicorp/aws"
      configuration_aliases = [aws.us_east_1]   # if module needs a named alias
    }
  }
}
```

**No `version = "~> 6.31"` here.** The caller pins the exact version. A module that pins its own provider version forces every caller to use that exact version, and they conflict.

The `configuration_aliases` field declares "this module expects a provider with this alias to be passed in" — without it, the caller can pass `providers = { aws.us_east_1 = ... }` but the module won't find it. Always declare aliases used inside the module.

### `variables.tf`

```hcl
variable "name" {
  description = "Name prefix for resources in this module."
  type        = string

  validation {
    condition     = can(regex("^[a-z0-9-]+$", var.name))
    error_message = "name must be lowercase alphanumeric with hyphens."
  }
}

variable "tags" {
  description = "Tags applied to every resource. Module merges its own tags in."
  type        = map(string)
  default     = {}
}

variable "subnets" {
  description = "Subnets to deploy into, keyed by AZ."
  type = map(object({
    cidr_block         = string
    availability_zone  = string
    map_public_ip      = optional(bool, false)
    nat_gateway        = optional(bool, false)
  }))
}

variable "engine_version" {
  description = "Database engine version."
  type        = string
  default     = "16.4"

  validation {
    condition     = startswith(var.engine_version, "16.")
    error_message = "Only Postgres 16.x is supported by this module."
  }
}
```

Rules:

- **Every variable has `description`.** It shows up in `terraform-docs`-generated READMEs and `tofu plan` UI. No exceptions.
- **Type every variable.** Even strings — write `type = string`. Untyped variables silently coerce, and `tflint` flags them.
- **Object types for structured input** (see `subnets` above) with `optional(type, default)` for fields that have sensible defaults. The 1.3+ `optional()` with a default is the right tool — no more nested ternaries to handle missing keys.
- **Use `validation {}` aggressively.** Multiple `validation` blocks per variable are supported. Validate enums, regex patterns, ranges. Fail at the boundary, not at the resource.
- **`sensitive = true`** on variables that should never appear in plan output.
- **Don't use `default = null`** to mean "optional" — use `optional()` inside an object type, or model it as an explicit "enabled" toggle.

### `outputs.tf`

```hcl
output "id" {
  description = "Identifier of the primary resource."
  value       = aws_db_instance.main.id
}

output "endpoint" {
  description = "Connection endpoint."
  value       = aws_db_instance.main.endpoint
}

output "security_group_id" {
  description = "Security group attached to the database."
  value       = aws_security_group.db.id
}
```

Rules:

- **Output anything a caller might plausibly need.** A missing output forces the caller to either fork the module or re-derive the value via data sources. Cheap to add, expensive to add later.
- **`description` on every output.** Same rule as variables.
- **`sensitive = true`** for anything that contains a password, token, or connection string with credentials.
- **`precondition {}` inside an output** to assert invariants before the caller consumes the value:
  ```hcl
  output "endpoint" {
    value = aws_db_instance.main.endpoint
    precondition {
      condition     = aws_db_instance.main.engine == "postgres"
      error_message = "Module is only configured for Postgres."
    }
  }
  ```

## Composability

A good module exposes the **outcome** (a VPC with these subnets) and hides the **implementation** (which AWS resources). A bad module exposes the implementation (an `aws_vpc` resource with all 30 arguments passed through). The first composes; the second is just indirection.

Concretely:

| Bad | Good |
|---|---|
| `variable "vpc_cidr_block"` on a "platform" module that wraps everything | Module accepts a `network` object that describes the network shape; module decides CIDR allocations internally |
| Module re-exports every attribute of every internal resource | Module exposes a small set of named outputs (`id`, `endpoint`, `arn`, `security_group_id`) |
| Module has a single argument that's a 200-key object | Either split into multiple modules, or this isn't a module — it's a config file |
| Module makes implementation decisions via boolean flags (`use_nat_gateway`, `use_transit_gateway`, `use_vpn`, …) | Module is opinionated and small; caller composes multiple narrow modules instead |

### Passing providers to a module

Modules **never** declare `provider {}` blocks. Pass providers explicitly when:

- The module needs an aliased provider (e.g. cross-region resources).
- The module needs a different provider than the caller's default.

```hcl
# caller (environments/aws/prod/main.tf)
provider "aws" {
  region = "us-west-2"
}

provider "aws" {
  alias  = "us_east_1"
  region = "us-east-1"
}

module "cdn" {
  source = "../../modules/cdn"

  providers = {
    aws           = aws            # default; passed explicitly for clarity
    aws.us_east_1 = aws.us_east_1  # ACM certs for CloudFront live in us-east-1
  }
}
```

The module declares the aliases it expects in `versions.tf` (`configuration_aliases = [aws.us_east_1]`). If you skip that declaration, the module silently uses the default provider and `aws.us_east_1` won't resolve.

### Provider `for_each` (OpenTofu only)

OpenTofu 1.9+ supports `provider "aws" { for_each = ... }`, which lets a single module fan out across regions or accounts without copy-pasted provider blocks. Terraform proper doesn't have this — for Terraform you generate provider blocks explicitly or use one provider per module instance.

See [opentofu.md](opentofu.md) for the full pattern.

## Sources

A module's `source` argument decides where it comes from. The five common forms:

| Source | Example | Use |
|---|---|---|
| Local path | `source = "../../modules/vpc"` | In-repo modules. Most common. |
| Git tag | `source = "git::https://github.com/org/tf-modules.git//vpc?ref=v1.4.0"` | Cross-repo modules. **Always pin to a tag or SHA.** |
| Terraform Registry | `source = "terraform-aws-modules/vpc/aws"` + `version = "~> 5.13"` | Public registry. Terraform-native. |
| OpenTofu Registry | `source = "github.com/opentofu/registry/tofu-aws-modules/vpc"` | OpenTofu uses a Git-based registry, not HashiCorp's |
| OCI artifact | `source = "oci://ghcr.io/org/modules/vpc:v1.4.0"` | OpenTofu 1.8+; modules published as OCI artifacts |

Never `ref=main`. Never `ref=HEAD`. A module is a dependency — pin it like one.

### Public registry modules: trust but verify

The `terraform-aws-modules/*` set on Terraform Registry is widely used and reasonable, but it's not magic. Pin to a specific minor version, read the changelog before bumping, and don't be afraid to fork or vendor when a module's defaults don't match your conventions. Same rule for any third-party module.

OpenTofu users: HashiCorp's public registry is **still accessible** from `tofu`, but as of 2024 OpenTofu also has its own registry. Both work. Pick one per repo and stay consistent.

## The "wrapper module" anti-pattern

Resist:

```hcl
# modules/s3-bucket/main.tf — WRAPPER MODULE
resource "aws_s3_bucket" "this" {
  bucket = var.bucket_name
  tags   = var.tags
}

resource "aws_s3_bucket_versioning" "this" {
  bucket = aws_s3_bucket.this.id
  versioning_configuration {
    status = var.versioning ? "Enabled" : "Suspended"
  }
}
```

This isn't a module — it's a slightly-smaller copy of the resource API. Callers gain nothing and lose the AWS provider's evolving features. **Just use the resource directly.** A bucket isn't worth a module unless you're encoding *your org's bucket conventions* — encryption, lifecycle, public-access block, replication, logging — into one opinionated wrapper that all buckets in your org use.

The rule: a module is justified by **opinion** (decisions baked in) and **reuse** (multiple call sites). A module with no opinions is documentation written as code.

## The "monolith module" anti-pattern

The opposite failure: one "platform" module that takes 80 inputs and provisions a VPC, EKS cluster, RDS, ALB, monitoring, IAM, and DNS, all gated by feature flags. Symptoms:

- Variables file is 800 lines.
- Half the module is `count = var.enable_x ? 1 : 0` patterns.
- A change to the database forces a plan against every other resource.
- Nobody knows what changes when you bump the version.

The fix: split it into 4–7 narrow modules (`network/`, `cluster/`, `database/`, etc.) and compose them at the environment level. The environment becomes the composition root; the modules become reusable parts.

## Versioning and breaking changes

For modules consumed across repos (git/registry sources):

- **Semver.** Bump major when you remove or rename an input/output, change a default in a way that causes drift, or change the resource graph in a way that triggers replacement. Bump minor for additive changes. Patch for fixes.
- **`moved {}` blocks** travel with the module — when you rename a resource internally, ship a `moved {}` block in the new version so callers' state migrates on apply. See [patterns.md](patterns.md).
- **Document upgrade paths** in the module's `CHANGELOG.md`. Bare minimum: which version requires which provider/Terraform version, and what `moved`/`removed` blocks the user might want to clean up after applying.

## Module testing

Write a `tofu test` run for every module that has 2+ call sites. The minimum useful test:

```hcl
# modules/vpc/tests/basic.tftest.hcl
run "plan_succeeds" {
  command = plan

  variables {
    name    = "test"
    cidr_block = "10.0.0.0/16"
  }

  assert {
    condition     = output.vpc_id != null
    error_message = "vpc_id should be set"
  }
}
```

See [testing.md](testing.md) for the full pattern, mocks, and when to escalate to `command = apply`.

## Don't / Do

| Don't | Do |
|---|---|
| Wrap a single resource in a module to "abstract it" | Use the resource directly; modules need opinions and reuse |
| `provider {}` block inside a module | Caller passes providers via `providers = { ... }` |
| Pin a provider `version =` inside a module's `required_providers` | Only `source`; caller pins the version |
| `source = "git::...?ref=main"` | `?ref=v1.4.0` or `?ref=<sha>` |
| Module exposes 80 inputs gated by flags | Split into smaller, opinionated modules |
| Module re-exports every internal attribute | Curate outputs: `id`, `endpoint`, `arn`, `security_group_id` |
| Untyped `variable` blocks | Always `type = ...`; always `description = ...` |
| `default = null` to mean "optional" | `optional(type, sensible_default)` inside an object type |
| Cross-stack data via `terraform_remote_state` | Native data sources (e.g. SSM Parameter, Secrets Manager) when available |
| `module "ec2_instance"` (type in the name) | `module "bastion"` (role in the name) |
