# Subagents

A **subagent** is a focused, single-purpose AI worker invoked by the main agent for a specific job (code review, exploration, planning, etc.). It runs with its own system prompt, tool allowlist, and optionally its own model. Outputs flow back to the parent as a single tool result.

The most common AI failure mode is making subagents too generic — "an agent that helps with the codebase". A good subagent has a tight scope (one job, one or two paragraphs of system prompt) and an aggressive description that delegates well. If you can't write a one-sentence trigger for it, the responsibility isn't carved cleanly.

## Where subagents live

Source of truth in this repo:

```
~/.dotfiles/ai/agents/<name>.md     # single-file format
```

Symlinked targets:

| Tool | Path | Format |
|---|---|---|
| Claude Code | `~/.claude/agents/<name>.md` | Markdown + YAML frontmatter (direct) |
| OpenCode | `~/.config/opencode/opencode.json` `agent.<name>` | JSON, generated by `generate-opencode-config.sh` from the markdown |
| Grok | `~/.grok/agents/<name>.md` | Markdown (inferred — confirm with `grok inspect`) |

OpenCode supports markdown agents natively too — both `~/.config/opencode/agent/<name>.md` and inline `agent: {...}` in `opencode.json` work. This repo uses the JSON-merge approach so a single markdown file feeds all three tools.

## Frontmatter — the unified schema

The shared schema below covers all three tools. Fields outside their tool's supported set are silently dropped on conversion.

```yaml
---
name: code-reviewer          # required everywhere; lowercase + hyphens
description: ...             # required; drives delegation (see below)
model: sonnet                # short name or full ID; see "Model" below

# Tool gating (use one or the other, not both):
tools: Read, Grep, Glob      # Claude Code: comma- or space-separated allowlist
                             # OpenCode: legacy boolean form; deprecated
disallowedTools: Write, Edit # Claude Code: denylist applied before allowlist

# OpenCode-specific:
mode: subagent               # primary|subagent|all (default: all)
permission:                  # NEW preferred form (replaces tools: in OpenCode)
  edit: ask                  # ask|allow|deny
  bash: { "*": ask, "git status": allow }
  write: deny
  webfetch: allow
temperature: 0.2             # 0.0-1.0
top_p: 0.9                   # 0.0-1.0
steps: 30                    # max agentic iterations (alias: maxSteps)
hidden: false                # hide from @ autocomplete
color: blue                  # or hex; rendering varies per tool

# Claude Code-specific (all optional):
permissionMode: default      # default|acceptEdits|auto|dontAsk|bypassPermissions|plan
maxTurns: 20
skills: [api-conventions]    # preload these skills' full bodies at startup
mcpServers: [github]         # by name (configured elsewhere) or inline defs
hooks: { ... }               # subagent-scoped hooks
memory: project              # user|project|local — persistent memory dir
background: false
effort: medium               # low|medium|high|xhigh|max
isolation: worktree          # run in a temp git worktree
initialPrompt: ...           # auto-submitted as first user turn (main-agent invocations only)
---
```

**Body**: the markdown after the frontmatter is the agent's system prompt. Keep it scoped — what's the agent's role, what does it focus on, what should it avoid. A few hundred words is usually plenty.

## The description is the delegation cue

When the parent agent decides whether to call a subagent (via the `Agent` tool, formerly `Task`), it reads the description. Same rules as skills:

1. State the job in one sentence.
2. Add "Use proactively when…" or "Use when the user asks…" with concrete triggers.
3. Include the file types or domains it covers.

Example:

```yaml
description: Reviews diffs for security and style issues. Use proactively after code changes to flag bugs, secrets, OWASP risks, or violations of project conventions.
```

## Model selection

| Value | Effect |
|---|---|
| `sonnet`, `opus`, `haiku` | Short alias — installer maps to `anthropic/claude-{model}-4-X` (current revision) for OpenCode |
| `claude-opus-4-7`, `anthropic/claude-sonnet-4-6`, etc. | Full provider ID, passed through verbatim |
| `inherit` | Claude Code — use the parent agent's model (default if field omitted) |
| omitted | Claude Code: inherit; OpenCode: falls back to global default |

The default for new subagents in this repo is **`sonnet`** unless you have a reason for opus (complex multi-step reasoning) or haiku (simple bulk work). The installer's `generate-opencode-config.sh` maps short names to full IDs; check the script for the current mapping if the model revision matters.

## Tools — the three styles

Three different tools, three different ways to express tool allowlists. This is the messiest part of subagent authoring.

### Claude Code (`tools:`)

Comma- or space-separated list of tool names. Omit to inherit everything. `*` is **not** documented.

```yaml
tools: Read, Grep, Glob, Bash, Edit
disallowedTools: Write     # denylist wins
```

Valid tool names include: `Read`, `Write`, `Edit`, `Bash`, `Grep`, `Glob`, `WebFetch`, `WebSearch`, `TodoWrite`, `Skill`, `Agent` (the renamed `Task` tool — old name aliases). MCP tools use the `mcp__server__tool` form. Restricting `Agent(worker, researcher)` allowlists which subagents this one can spawn.

### OpenCode (`permission:` preferred, `tools:` deprecated)

The new form is `permission:` — a per-tool map of `ask | allow | deny`:

```yaml
permission:
  edit: deny
  bash:
    "*": ask
    "git status": allow
    "git diff *": allow
  read: allow
  webfetch: deny
```

Keys: `read`, `edit`, `glob`, `grep`, `list`, `bash`, `task`, `external_directory`, `todowrite`, `webfetch`, `websearch`, `lsp`, `skill`, `question`, `doom_loop`.

The legacy `tools: { read: true, bash: false }` form still works but is marked `@deprecated` in source. The `generate-opencode-config.sh` script in this repo currently emits the deprecated form — that's a known gap.

### Grok Build

Same schema as Claude Code (compat). Confirm with `grok inspect <name>`.

## Portability — system prompts must be tool-agnostic

Same principle as skills: subagents in `~/.dotfiles/ai/agents/` are installed into all three tools. The markdown body becomes the agent's system prompt regardless of which tool is running it. Write it for "the agent", not for a specific brand.

| Do this | Not this |
|---|---|
| "Read files with the Read tool" | "Use Claude Code's Read tool" |
| "Run shell commands via Bash" | "Use the `bash` tool in OpenCode" |
| "If you need to delegate, spawn a subagent" | "Use the `Agent` tool" |

Tool-specific **frontmatter** is fine — `tools: Read, Grep` (Claude shape) and `permission: { edit: deny }` (OpenCode shape) can coexist. Each tool ignores keys it doesn't understand. Just keep the **prompt body** universal.

If the agent legitimately needs to behave differently in one tool, that's a sign it should be two agents, not one with conditional logic in the system prompt.

## Body — the system prompt

The markdown body becomes the agent's system prompt. Guidelines:

- **Define the role in one sentence.** "You are a senior code reviewer focused on security."
- **State the scope.** What does it touch? What's out of scope?
- **Give a workflow.** Three to five numbered steps.
- **Define the output shape.** Markdown report? JSON? Pass/fail verdict?

Example body:

```markdown
You are a senior security reviewer. Your job is to read code changes and flag bugs,
secrets, and OWASP top-10 risks before they ship.

Workflow:
1. Run `git diff origin/main..HEAD` to see the change.
2. For each modified file, read it in full (not just the diff).
3. Flag: secret leakage, SQLi, XSS, deserialization, missing auth, command injection.
4. Output a markdown report with severity (HIGH/MED/LOW) and file:line refs.

Out of scope: style, formatting, test coverage — that's other reviewers' jobs.
```

## Invocation paths

| Tool | How parents invoke |
|---|---|
| Claude Code | Natural language ("Use the code-reviewer subagent…"), `@agent-<name>` mention, `Agent` tool with `subagent_type: <name>`, or `claude --agent <name>` to start a whole session as the subagent |
| OpenCode | `@<name>` in chat, or programmatically via the `task` tool. Primary agents (`mode: primary` or `all`) cycle via Tab |
| Grok | Same as Claude (via compat). `/agents` modal to manage |

## Plugin caveats (Claude Code)

If a subagent ships as part of a plugin, Claude Code **ignores** the following frontmatter fields for security: `hooks`, `mcpServers`, `permissionMode`. User and project subagents (which is what this repo produces) don't have this restriction.

## OpenCode mode field

`mode` controls how the agent surfaces in the OpenCode UI:

| Mode | Behavior |
|---|---|
| `primary` | User-facing top-level agent (Tab cycles between primary agents) |
| `subagent` | Only invoked by other agents via `@name` or the `task` tool |
| `all` (default) | Both — primary surface AND callable as subagent |

For agents you intend to use as helpers (code-reviewer, explorer, planner), set `mode: subagent`. For agents you'd switch to as a "session mode", set `mode: primary` or `all`.

## Minimal canonical example

See [`examples/agent.template.md`](examples/agent.template.md). Compact form:

```markdown
---
name: code-reviewer
description: Reviews diffs for security and style. Use proactively after code changes.
model: sonnet
tools: Read, Grep, Glob, Bash
mode: subagent
permission:
  edit: deny
  bash: { "*": ask }
---

You are a senior code reviewer. Read the diff, focus on correctness and security,
output a short markdown report with file:line refs.
```

## Editing an existing agent

When you update `~/.dotfiles/ai/agents/<name>.md`:

1. Save the file.
2. Run `dot update` — re-links all tools and regenerates OpenCode agent JSON. Or `dot install opencode` for OpenCode only.
3. Verify with `grok inspect <name>` (Grok) or by opening the `/agents` modal in Claude Code.

The JSON regeneration is non-destructive: `generate-opencode-config.sh` only touches managed keys (`.agent`, `.command`, `.instructions`, `.plugin`) and preserves your personal config in `opencode.json`.

## Common mistakes

- **Tools field on a subagent in OpenCode.** Use `permission:` instead — the `tools: { ... }` form is deprecated and may be removed.
- **`name` field that doesn't match the filename.** The filename is what users see; the `name` field is what the model uses for invocation. Keep them in sync.
- **Model: `*` or `any`.** Not supported anywhere. Use `inherit` (Claude) or omit (everywhere) to defer to the caller.
- **Inline MCP server definitions in agent frontmatter.** This works in Claude Code but doesn't propagate to OpenCode. Configure MCP servers centrally via `ai/mcp-servers.json.tpl` and reference them by name.
- **Long system prompts.** A 2,000-word system prompt is usually a sign the agent is doing too much. Split into two agents or move the depth into a referenced skill.
